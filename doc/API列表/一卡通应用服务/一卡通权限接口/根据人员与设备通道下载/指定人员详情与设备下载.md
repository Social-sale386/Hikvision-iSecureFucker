# 指定人员详情与设备下载

> 接口说明
> 针对少量权限（100设备通道*1000人）的下发场景，以异步任务方式下发出入控制权限。
> 1、接口支持下发优先级设置。优先级分为正常与快速两种方式，默认优先级为正常，其他异步下发接口的优先级都为正常方式。高优先级方式下发适用于需要权限快速生效的场景，但是频繁的使用高优先方式会造成下发队列的频繁切换可能会造成下发性能下降，请根据业务场景选择。
> 2、 每个tagId全局默认最多保持1000个可操作（未下发结束）任务列表。
> 说明：调用方在使用该接口下发权限时，如果也使用权限配置功能，那么可能会造成人员权限冲突的情况，acps以最后一次操作数据为准；

## 基本信息
|项目|值|
|---|---|
|接口版本|v1|
|适配版本|综合安防管理平台iSecure Center 1.5.0及以上版本|
|接口地址|`/api/acps/v1/authDownload/special/person/diy`|
|请求方法|`POST`|
|数据提交方式|application/json|

## 快速调用
```text
POST /api/acps/v1/authDownload/special/person/diy
```

## 请求参数
|参数|位置|类型|必填|说明|
|---|---|---|---|---|
|`priority`|`body`|`integer`|否|优先级:|
|`resourceInfos`|`body`|`object[]`|是|资源对象列表|
|`+resourceIndexCode`|`Body`|`string`|是|资源的唯一标识，资源为设备，查询资源列表v2接口获取返回报文中的indexCode字段|
|`+resourceType`|`Body`|`string`|是|资源类型，详见附录A.50 设备通道类型|
|`+channelNos`|`Body`|`number[]`|否|资源通道号， 当资源类型为设备时，无特殊说明必填。 传参数说明： 表示此次数据需要配置/下载到设备的哪些通道上； 门禁的人证设备、可视对讲设备等无通道的设备，默认通道号为1；|
|`personInfos`|`body`|`object[]`|是|人员对象列表|
|`+personId`|`Body`|`string`|是|人员编号列表。|
|`+personStatus`|`Body`|`string`|是|操作类型（ADD新增；UPDATE修改；DELETE删除）|
|`+startTime`|`Body`|`string`|否|开始日期，配置人员权限的有效期，为空时默认长期有效。采用ISO8601时间格式，最大长度32个字符，如2019-09-03T17:30:08.000+08:00|
|`+endTime`|`Body`|`string`|否|结束日期，配置人员权限的有效期，为空时默认长期有效。采用ISO8601时间格式，最大长度32个字符，如2019-09-03T17:30:08.000+08:00|
|`+name`|`Body`|`string`|否|人员姓名|
|`+personType`|`Body`|`string`|是|人员类型(COMMON普通人GUEST访客BLACKLIST黑名单SUPER管理员DISABLED行动不便人员)|
|`+cardInfo`|`Body`|`object`|否|卡片信息列表|
|`++cardList`|`Body`|`object[]`|否|人员卡片集合(最多十张)|
|`+++cardNo`|`Body`|`string`|是|卡号，支持8-20位数字+大写字母的卡号；请根据设备实际能力填写卡号|
|`+++cardType`|`Body`|`string`|是|卡类型，normalCard普通卡 hijackCard胁迫卡 patrolCard巡更卡）|
|`+++cardStatus`|`Body`|`string`|是|状态（ADD新增；UPDATE修改；DELETE删除）|
|`+faceInfo`|`Body`|`object`|否|人脸信息|
|`++deleteAllFace`|`Body`|`boolean`|否|是否清除人员下所有人脸|
|`++cardNo`|`Body`|`string`|否|卡号：支持8-20位数字+大写字母的卡号；指纹绑定卡片（以卡为中心设备必填），请根据设备实际能力填写|
|`++storageNodeCode`|`Body`|`string`|否|asw图片服务器存储节点编号，将人脸图片数据存储在asw时，可填写图片服务器的存储节点编号，并在人脸数据data的value中填写asw返回的url（无ip端口）|
|`++faceList`|`Body`|`object[]`|是|人员人脸集合(deleteAllFace为true时,有值系统不处理)|
|`+++faceId`|`Body`|`string`|是|人脸编号|
|`+++picUrl`|`Body`|`string`|是|人脸数据|
|`+personExtendData`|`Body`|`object`|否|人员扩展参数|
|`++personProperty`|`Body`|`number`|否|人员特性， 1：普通人，默认类型； 2：来宾，来宾人员比对通过或者时，会上报来宾事件，可正常开门； 2：黑名单，黑名单人员比对通过时，会上报黑名单事件，且无法开门； 4:超级用户，为需要超级权限的用户而设置。不受限于门常闭、反潜回、刷卡+密码认证需要密码确认的规则，刷卡直接开门，但要遵循多门互锁及多重认证规则。|
|`++userVerifyMode`|`Body`|`string`|否|人员验证方式， 人员验证方式的优先级高于读卡器验证方式，需要根据设备能力集判断，设备仅支持部分字段）：cardAndPw-刷卡+密码，card-刷卡，cardOrPw-刷卡或密码，fp-指纹，fpAndPw-指纹+密码，fpOrCard-指纹或刷卡，fpAndCard-指纹+刷卡，fpAndCardAndPw-指纹+刷卡+密码，faceOrFpOrCardOrPw-人脸或指纹或刷卡或密码，faceAndFp-人脸+指纹，faceAndPw-人脸+密码，faceAndCard-人脸+刷卡，face-人脸， fpOrPw-指纹或密码， faceAndFpAndCard-人脸+指纹+刷卡，faceAndPwAndFp-人脸+密码+指纹， faceOrfaceAndCard-人脸或人脸+刷卡，fpOrface-指纹或人脸，cardOrfaceOrPw-刷卡或人脸或密码，cardOrFace-刷卡或人脸，cardOrFaceOrFp刷卡或人脸或指纹（默认设备本身的认证方式，仅支持以人为中心设备）|
|`++maxOpenDoorTime`|`Body`|`number`|否|最大认证次数，当且仅当personProperty是来宾时有效 0为无次数限制（默认为0）|
|`++leaderPersonEnabled`|`Body`|`boolean`|否|是否是首人，需要在设备上配置首卡开门持续时长|
|`++closeDelayEnabled`|`Body`|`boolean`|否|是否关闭延迟开门，需在设备上配置延迟开门时长|

补充说明:
- 1正常下载，将下载任务的按先后顺序加入下载队列排队下载
- 2快速下载，将下载任务插入下载队列的头部插队下载
- 内部人员可通过查询人员列表v2接口获取返回报文中的personId字段。
- 外部人员标识最大长度64位，设备仅支持32位，超过时将截取前32位。

## 返回参数
|参数|位置|类型|必填|说明|
|---|---|---|---|---|
|`code`|`body`|`string`|是|返回码，0 – 成功，其他- 失败，参考附录E.2.3 出入控制权限错误码|
|`msg`|`body`|`string`|否|返回描述|
|`data`|`body`|`string`|否|无返回结果|

## 返回参数示例
```json
{
  "code": "0",
  "msg": "success"
}
```
