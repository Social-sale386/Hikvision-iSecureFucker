# 查询单个权限条目

> 接口说明
> 根据人员和设备通道查询一条权限条目信息，适用于判断指定人员与通道的权限情况，比如中心控制，远程开门等场景。
> V1.5.0版本返回字段新增personExtendData，表示人员的权限扩展参数

## 基本信息
|项目|值|
|---|---|
|接口版本|v1|
|适配版本|综合安防管理平台iSecure Center V1.3及以上版本|
|接口地址|`/api/acps/v1/auth_item/single/search`|
|请求方法|`POST`|
|数据提交方式|application/json|

## 快速调用
```text
POST /api/acps/v1/auth_item/single/search
```

## 请求参数
|参数|位置|类型|必填|说明|
|---|---|---|---|---|
|`personId`|`body`|`string`|是|人员ID，查询人员列表v2接口获取返回报文中的personId字段|
|`resourceInfo`|`body`|`object`|是|设备通道对象，只支持单个通道查询|
|`+resourceIndexCode`|`Body`|`string`|是|资源的唯一标识，资源为设备，查询资源列表v2接口获取返回报文中的indexCode字段|
|`+resourceType`|`Body`|`string`|是|资源类型，详见附录A.50 设备通道类型|
|`+channelNos`|`Body`|`number[]`|否|通道号|

## 请求参数示例
```json
{
  "personId": "02e66ac47a03481fa0f304fa3acfa773",
  "resourceInfo": {
    "resourceIndexCode": "0a869b06f4ff4da29949acf006405c39",
    "resourceType": "acsDevice",
    "channelNos": [
      1
    ]
  }
}
```

## 返回参数
|参数|位置|类型|必填|说明|
|---|---|---|---|---|
|`code`|`body`|`string`|否|返回码，0 – 成功，其他- 失败，参考附录E.2.3 出入控制权限错误码|
|`msg`|`body`|`string`|否|返回描述|
|`data`|`body`|`object`|否|返回数据|
|`+personId`|`body`|`string`|否|人员编号|
|`+resourceIndexCode`|`body`|`string`|否|资源的唯一标识|
|`+channelNo`|`body`|`number`|否|通道号|
|`+channelIndexCode`|`body`|`string`|否|资源通道唯一标识（在权限下载和配置接口入参时对应查询接口会返回，不作为查询接口的查询条件）|
|`+cardJson`|`body`|`string`|否|记录已下载到设备的卡号（Map格式（卡号：状态），其中状态(0改变表示下载后卡片的信息又变化了，1已下载)）|
|`+templateConfigInfos`|`body`|`object[]`|否|计划模板|
|`++dimension`|`body`|`number`|否|维度,范围：1-4，支持4维度计划模板配置|
|`++templateId`|`body`|`string`|否|计划模板Id,默认全天候计划模板为1。计划模板详细信息可通过计划模板管理接口添加。|
|`+personStatus`|`body`|`number`|否|人员状态,|
|`+cardStatus`|`body`|`number`|否|卡片状态,|
|`+faceStatus`|`body`|`number`|否|人脸状态,|
|`+configTime`|`body`|`string`|否|配置时间|
|`+startTime`|`body`|`string`|否|人员权限有效期（以人为中心设备时使用）,开始时间|
|`+endTime`|`body`|`string`|否|人员权限有效期（以人为中心设备时使用）,结束时间|
|`+personExtendData`|`body`|`object`|否|人员扩展参数，|
|`++personProperty`|`body`|`number`|否|人员特性， 1：普通人，默认类型； 2：来宾，来宾人员比对通过或者时，会上报来宾事件，可正常开门； 2：黑名单，黑名单人员比对通过时，会上报黑名单事件，且无法开门； 4:超级用户，为需要超级权限的用户而设置。不受限于门常闭、反潜回、刷卡+密码认证需要密码确认的规则，刷卡直接开门，但要遵循多门互锁及多重认证规则。|
|`++userVerifyMode`|`body`|`string`|否|人员验证方式， 人员验证方式的优先级高于读卡器验证方式，需要根据设备能力集判断，设备仅支持部分字段）：cardAndPw-刷卡+密码，card-刷卡，cardOrPw-刷卡或密码，fp-指纹，fpAndPw-指纹+密码，fpOrCard-指纹或刷卡，fpAndCard-指纹+刷卡，fpAndCardAndPw-指纹+刷卡+密码，faceOrFpOrCardOrPw-人脸或指纹或刷卡或密码，faceAndFp-人脸+指纹，faceAndPw-人脸+密码，faceAndCard-人脸+刷卡，face-人脸， fpOrPw-指纹或密码， faceAndFpAndCard-人脸+指纹+刷卡，faceAndPwAndFp-人脸+密码+指纹， faceOrfaceAndCard-人脸或人脸+刷卡，fpOrface-指纹或人脸，cardOrfaceOrPw-刷卡或人脸或密码，cardOrFace-刷卡或人脸，cardOrFaceOrFp刷卡或人脸或指纹（默认设备本身的认证方式，仅支持以人为中心设备）|
|`++maxOpenDoorTime`|`body`|`number`|否|最大认证次数，当且仅当personProperty是来宾时有效 0为无次数限制（默认为0）|
|`++leaderPersonEnabled`|`body`|`boolean`|否|是否是首人，需要在设备上配置首卡开门持续时长|
|`++closeDelayEnabled`|`body`|`boolean`|否|是否关闭延迟开门，需在设备上配置延迟开门时长|
