# 按人员设置人员权限扩展参数

> 接口说明
> 权限配置支持按人人员和设备通道配置权限，适用人员权限有特殊参数需要下发的场景。说明：权限配置数据采用异步分批入库方式，接口调用成功后返回权限配置单编号，可通过《查询权限配置进度》，在配置的过程中分批插入数据，只有当配置单结束时才能查询到完整的权限配置信息。
> 权限配置的预计时间：每10万配置数据大概30秒。（配置数据量：人员数据量*设备数据量）。
> 1、 相同的配置数据重复配置时，第一次配置生效后，后面相同的配置将自动过滤丢弃。
> 2、 在相同人员和设备通道多次配置时，权限有效期和计划模板信息以最后一次配置为准（比如：先对人员分组A与设备分组D1，D2配置了计划模板T1，有效期为day1的权限，再对人员分组A，B与设备分组D1配置计划模板t2，有效期为day2的权限，那么人员分组A与设备分组D1存在重复配置情况，最终人员分组A与设备分组D1的计划模板为t2，有效期为day2）；
> 3、 人员扩展参数变更后，相同的参数以最后一次为准，不同的参数会和以前的参数合并，第一次需要传入人员扩展参数初始化值，供权限删除时，设置设备上为初始化值。
> 4、 tagId用于让多个应用共用出入控制权限服务时，用以区分各自的配置信息。建议使用组件标识。注意点：不同业务组件设备通道隔离，应该根据业务场景使用不同的设备通道配置权限；如对相同的设备通道都有业务应用，那么人员隔离，应该根据场景使用不同的人员，否则会造成权限条目归属相互竞争的情况，在权限条目综合查询时，数据归属以最后一次入库配置为准。

## 基本信息
|项目|值|
|---|---|
|接口版本|v1|
|适配版本|综合安防管理平台iSecure Center V1.5.0及以上版本|
|接口地址|`/api/acps/v1/authConfig/configuration/extend`|
|请求方法|`POST`|
|数据提交方式|application/json|

## 快速调用
```text
POST /api/acps/v1/authConfig/configuration/extend
```

## 请求参数
|参数|位置|类型|必填|说明|
|---|---|---|---|---|
|`personIds`|`body`|`string[]`|否|人员ID列表，查询人员列表v2接口获取返回报文中的personId字段|
|`resourceInfos`|`body`|`object[]`|否|资源设备对象列表|
|`+resourceIndexCode`|`Body`|`string`|否|资源的唯一标识，资源为设备，查询资源列表v2接口获取返回报文中的personId字段|
|`+resourceType`|`Body`|`string`|否|资源类型，详见附录A.50 设备通道类型|
|`+channelNos`|`Body`|`number[]`|否|资源通道号|
|`startTime`|`body`|`string`|否|开始日期，与结束日期要么都为空要么都不为空。配置人员权限的有效期，为空时默认长期有效。采用ISO8601时间格式，最大长度32个字符，如2019-09-03T17:30:08.000+08:00|
|`endTime`|`body`|`string`|否|结束日期，与开始日期要么都为空要么都不为空。配置人员权限的有效期，为空时默认长期有效。采用ISO8601时间格式，最大长度32个字符，如2019-09-03T17:30:08.000+08:00|
|`personExtendData`|`body`|`object`|是|人员扩展参数，|
|`+personProperty`|`Body`|`number`|否|人员特性，|
|`+userVerifyMode`|`Body`|`string`|否|人员验证方式，|
|`+maxOpenDoorTime`|`Body`|`number`|否|最大认证次数，当且仅当personProperty是来宾时有效|
|`+leaderPersonEnabled`|`Body`|`boolean`|否|是否是首人，需要在设备上配置首卡开门持续时长|
|`+closeDelayEnabled`|`Body`|`string`|否|是否关闭延迟开门，需在设备上配置延迟开门时长|

补充说明:
- 当删除时扩展时：value是整形则下发0，字符串则下发空字符串，数组则下发空数组，布尔型则下发flase。
- 1：普通人，默认类型；
- 2：来宾，来宾人员比对通过或者时，会上报来宾事件，可正常开门；
- 2：黑名单，黑名单人员比对通过时，会上报黑名单事件，且无法开门；
- 4:超级用户，为需要超级权限的用户而设置。不受限于门常闭、反潜回、刷卡+密码认证需要密码确认的规则，刷卡直接开门，但要遵循多门互锁及多重认证规则。
- 人员验证方式的优先级高于读卡器验证方式，需要根据设备能力集判断，设备仅支持部分字段）：cardAndPw-刷卡+密码，card-刷卡，cardOrPw-刷卡或密码，fp-指纹，fpAndPw-指纹+密码，fpOrCard-指纹或刷卡，fpAndCard-指纹+刷卡，fpAndCardAndPw-指纹+刷卡+密码，faceOrFpOrCardOrPw-人脸或指纹或刷卡或密码，faceAndFp-人脸+指纹，faceAndPw-人脸+密码，faceAndCard-人脸+刷卡，face-人脸， fpOrPw-指纹或密码， faceAndFpAndCard-人脸+指纹+刷卡，faceAndPwAndFp-人脸+密码+指纹， faceOrfaceAndCard-人脸或人脸+刷卡，fpOrface-指纹或人脸，cardOrfaceOrPw-刷卡或人脸或密码，cardOrFace-刷卡或人脸，cardOrFaceOrFp刷卡或人脸或指纹（默认设备本身的认证方式，仅支持以人为中心设备）
- 0为无次数限制（默认为0）

## 请求参数示例
```json
{
  "resourceInfos": [
    {
      "resourceIndexCode": "ef72fd5e7ec341a99a37ecd99f90af6e",
      "resourceType": "acsDevice",

        1
      ]
    }
  ],
  "personIds": [
    "37835310 - f786 - 451 a - 8e b5 - 5 d6c43b827cc"
  ],
  "personExtendData": {
    "userVerifyMode": "faceAndCard",
    "personProperty": 2
  }
```

## 返回参数
|参数|位置|类型|必填|说明|
|---|---|---|---|---|
|`code`|`body`|`string`|否|返回码，0 – 成功，其他- 失败，参考附录E.2.3 出入控制权限错误码|
|`msg`|`body`|`string`|否|返回描述|
|`data`|`body`|`object`|否|配置单唯一编号|
|`+taskId`|`body`|`string`|否|配置单唯一编号|

## 返回参数示例
```json
{
  "code": "0",
  "msg": "success",
  "data": {
    "taskId": "317735290e9047898f6bb2f9aa0ec745"
  }
}
```
